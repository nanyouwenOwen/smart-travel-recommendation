# 规划—实施—审核与测试工作模式

## 强制流程

每次非简单修改先创建 `docs/plans/round-XX-<topic>.md`，明确目标、非目标、契约、文件级步骤、测试和完成定义。规划终端只写计划，不修改实现。

实施终端严格按计划修改代码，优先先改 OpenAPI/接口约束，再改后端、前端、迁移和测试；过程中记录真实失败和修正，不把未运行的命令写成“通过”。

独立审核终端不得修改实现，只在 `docs/reviews/` 写报告。审核必须比对计划、TODO、实际 diff 和运行证据，且只能给出 `PASS` 或 `FAIL`；存在任何阻断就是 `FAIL`，非阻断观察项可在 `PASS` 报告中单列。`FAIL` 时列出最小阻断、位置和复审条件。实施终端修复后，由同一审核终端复审。

只有本地门禁、本次提交对应的 GitHub Actions required jobs 和独立审核均通过，才能勾选 TODO、提交里程碑和更新交接记录；旧提交的绿色 run 不能代替当前修改证据。涉及 tag、Release、公开部署、生产 secret、生产数据或第三方付费资源，必须停下来取得负责人明确授权。

## 最低验证矩阵

- 契约：固定版本 OpenAPI lint。
- 前端：Prettier、ESLint、类型检查、覆盖率、生产构建；交互变更增加 Playwright。
- 后端：Spotless、SpotBugs、`mvn verify`；迁移变更必须经 MySQL 8.4。
- 容器/运维：Compose config、健康、非 root、备份恢复和恢复 smoke。
- 安全：依赖/secret 扫描、鉴权和资源归属回归。
- 交付：候选产物、SBOM、SHA256、Git SHA 和 CI run 可追溯。

小改动可缩短计划，但不能由实施者自审后直接宣称完成。紧急修复也应事后补齐计划、审核和验证记录。
