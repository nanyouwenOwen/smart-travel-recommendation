# Round 08：Compose Smoke 确定性与失败诊断计划

## 1. 背景与问题界定

纯文档提交 `3fe06a6` 对应的 GitHub Actions run `29164981696` 在 `container-smoke` job 的 `compose-smoke.sh` 步骤运行约 2 分 32 秒后失败，依赖它的 `release-candidate` 被跳过；紧邻、业务代码相同的提交 `82d0f89` 对应 run `29164801003` 全部通过。公共未认证页面目前不能提供失败步骤末尾的完整日志，因此不能在缺少证据时把故障归因到某一条具体命令。

现有 `scripts/compose-smoke.sh` 把启动、业务 smoke、备份恢复、性能、后端重启及 MySQL 恢复串在一个紧凑脚本中，并由 `set -euo pipefail` 控制退出。主要可靠性风险是：轮询体内的瞬时连接失败或非 2xx 响应缺少上下文；MySQL stop/start 后后端可能先恢复 TCP/HTTP 响应、随后因连接池重连短暂返回 4xx/5xx；最终登录直接用 `curl -f` 判断，无法区分瞬时基础设施状态和稳定的业务断言失败。性能门也缺少明确阶段边界，容易使读者误判实际失败位置。

本轮只提高 smoke 门禁的可诊断性和对瞬时基础设施状态的容忍度，不更改产品功能、API 契约、业务断言、性能阈值、容器健康定义或发布授权边界。

## 2. 目标与非目标

目标：

1. 每个 smoke 阶段输出带时间的明确开始、成功或失败信息，CI 日志能够直接定位最后通过和首个失败阶段。
2. 轮询命令自行接住预期的瞬时 `curl`/容器状态失败，不让 `set -e` 在重试窗口内提前终止。
3. MySQL 恢复验证等待“后端能够完成一次有效登录”，而不是仅等待端口或健康端点短暂响应；超时后输出 HTTP 状态、响应摘要和容器诊断。
4. 重试仅覆盖启动、重启、数据库恢复和异步行程生成等明确的瞬时状态；注册、鉴权内容、备份数据、非 root、SSE 终态和性能阈值等断言保持严格。
5. 性能阶段有独立日志和退出码传播，性能失败不能被后续恢复测试掩盖，基础设施失败也不能被误报为性能失败。
6. 同一独立审核者复核实现与重跑结果；本轮提交对应的 required CI jobs 全绿后才结束。

非目标：

- 不通过重复执行整个 job、无条件 `|| true`、放宽 k6 阈值或延长所有超时来制造绿色结果。
- 不在失败后自动重试注册、创建行程、SSE、备份内容等非幂等或业务正确性断言。
- 不修改应用健康检查、数据库连接池、Compose 服务定义或 GitHub Actions job 依赖，除非实施审查证明脚本内无法提供必要诊断；此类扩展必须先回到计划评估。
- 不创建 tag、Release 或部署，不勾选 `TODO.md` 的“发布 MVP”。

## 3. 实施方案

### 3.1 建立可读的阶段与失败框架

在 `scripts/compose-smoke.sh` 中增加小型 shell helper，而不是引入额外运行时依赖：

- `log`：输出 UTC 时间与阶段消息。
- `fail`：输出明确错误并以非零退出。
- `diagnostics`：失败时以 best-effort 方式打印 `docker compose ps -a`、各服务 health/status，以及 backend/mysql/frontend 的有界末尾日志；诊断命令不得覆盖原始失败退出码，也不得输出环境变量或凭据。
- 扩展现有 trap：记录原始退出码；仅在非零时先采集诊断，再执行删除临时备份和 `compose down -v`，最后保留原始退出码。信号退出同样保持清理。

把当前单行串联命令拆成具名阶段，至少包括：构建启动与健康、代理健康、非 root、注册、地点与行程 READY、SSE、备份恢复、性能、后端重启恢复、MySQL stop/start 恢复。日志不得回显 Bearer token、密码、完整认证响应或包含凭据的命令跟踪。

### 3.2 提供受控轮询 helper

实现一个通用的有限轮询结构（或职责清晰的少量专用 helper），要求：

- 明确最大次数、间隔和所等待的条件；每次失败在函数条件上下文中处理，使 `set -e` 不会因预期瞬时失败退出。
- 成功必须由条件函数返回 0；达到截止时间必须 `fail`，并报告最后观察到的状态。
- 不使用 `exit 0` 从轮询内部提前结束整个脚本，确保阶段成功日志、trap 和后续控制流一致。
- 只保留最后一个脱敏响应片段用于超时诊断；临时响应文件由 trap 删除。

适用位置：三服务初始 healthy、行程异步变为 `READY`、后端 restart 后健康恢复、MySQL restart 后登录恢复。初始服务最终状态仍必须逐个严格断言为 `healthy`。

### 3.3 收紧 HTTP 请求与业务断言

对一次性业务请求继续失败即退出，但提升错误可见性：

- 使用临时文件分离响应体与 HTTP 状态，或使用不会泄露请求头的封装；失败消息指出 endpoint/阶段和状态码。
- 注册响应必须验证非空 access token；地点搜索必须验证有效 `locationId`；创建行程必须验证有效 id；轮询完成必须严格等于 `READY`，不能把 JSON 解析错误视为可接受成功。
- SSE 必须继续验证 `done` 事件，且管道任一端失败均失败。
- 备份必须通过 gzip 校验，恢复库三项数据断言保持原阈值，非 root 断言不放宽。

异步行程 GET 中的瞬时网络错误可在既有时间窗内重试；稳定返回的终态失败、畸形 JSON 或超时均应以非零退出并给出最后状态。

### 3.4 修正重启恢复验证

后端 restart 后：

- 轮询 `/api/v1/health`，瞬时连接失败或非 2xx 只在限定窗口内重试。
- 超时后报告最后 HTTP 状态并触发统一诊断；成功后再记录阶段完成。

MySQL stop/start 后：

- 不把后端“短暂能响应”当成恢复完成；轮询现有用户的登录请求。
- `curl` 的连接错误、超时、HTTP 4xx/5xx、后端连接池尚未重建以及临时非 JSON 响应均在限定恢复窗口内视为待恢复状态，并记录最后 HTTP/传输结果。
- 只有 HTTP 2xx 且响应 JSON 中存在非空 access token 才算恢复成功。这样 401、空体或错误页不会被误判为恢复。
- 重试只适用于数据库重启这一明确的基础设施恢复窗口；窗口耗尽仍失败，不降低登录语义。

### 3.5 隔离性能门结果

在调用 `scripts/perf.sh` 前后输出独立阶段日志，并显式保留其退出状态：

- k6 任一阈值或执行失败立即以“性能阶段失败”结束，不能继续到重启阶段后被后一个结果覆盖。
- 不改变 `tests/performance/smoke.js` 中的阈值、请求数量或 `scripts/perf.sh` 的网络模式。
- 运行性能测试前确认当前代理/API 健康；该检查失败应标为性能前置基础设施失败，而不是 k6 阈值失败。

## 4. 安全与兼容约束

1. 保持 `set -euo pipefail`；只在有限轮询和 best-effort 诊断的条件上下文中显式接住非零状态。
2. 不启用 `set -x`；诊断禁止打印 `$header`、access token、密码、认证响应全文、Compose 环境或可能含密钥的 inspect 配置。
3. 诊断日志必须有界，避免 CI 日志失控；优先 `compose ps -a`、health 状态和每服务有限行数的末尾日志。
4. cleanup 在成功与失败路径都删除临时响应、SQL 备份、容器和卷；诊断失败不能阻断 cleanup。
5. 保持当前 Bash/Docker Compose/GNU runner 兼容性，不额外要求本地未声明工具。

## 5. 验证矩阵

### 5.1 静态和本地验证

实施者至少执行：

```bash
bash -n scripts/compose-smoke.sh scripts/perf.sh
git diff --check
docker compose config >/dev/null
bash scripts/compose-smoke.sh
```

并审阅实际输出，确认每一阶段有明确边界、成功路径退出 0、日志不包含 token/密码。若运行环境允许，再连续运行 smoke 至少两次，以验证独立 project/volume 清理和重跑稳定性；连续通过是补充证据，不替代断言审查。

### 5.2 定向失败注入

在不提交测试凭据或永久改变脚本的前提下，使用可恢复方式验证以下路径，记录命令/现象到审核报告：

1. 启动期间服务尚未 ready：轮询继续而非被 `set -e` 提前终止。
2. MySQL stop/start 后登录先返回连接失败或 5xx：在限定窗口内继续，恢复后以有效 token 通过。
3. 不可恢复条件（例如停止数据库且不启动，或使用临时错误 base/port）：达到截止时间后非零退出，显示最后状态与有界诊断，并完成资源清理。
4. 性能脚本模拟非零退出或临时收紧阈值：日志明确指向性能阶段且后续恢复阶段不执行；随后恢复所有临时改动，确认 `git status` 无残留。

失败注入不得为了验证而提交弱化断言或专用后门；若 CI 环境无法安全复现某一项，审核报告应注明替代的代码路径审查证据。

### 5.3 回归门禁

脚本定向验证通过后执行仓库完整门禁：

```bash
bash scripts/check.sh
```

确认业务/前后端测试未受影响，并检查 `git status` 不包含 `.env`、备份、响应文件、测试产物或其他用户文件。

## 6. 独立审核与修正循环

独立 reviewer 不修改实现，只创建 `docs/reviews/round-08-compose-smoke-reliability-review.md`，并完成：

1. 将实际 diff 逐条映射到本计划，确认重试范围只覆盖瞬时基础设施状态，没有 `|| true` 吞掉关键失败、全 job 重试或阈值放宽。
2. 审查所有 `set -e`/pipeline/函数/陷阱控制流，确认原始退出码保留、诊断先于清理且诊断失败不遮蔽根因。
3. 复核 MySQL 恢复成功条件必须同时满足 HTTP 2xx 和有效 token；4xx/5xx 只能有限重试，不能最终通过。
4. 复核性能非零立即失败且阶段命名准确；业务、SSE、恢复数据、非 root 和健康断言不弱化。
5. 运行静态检查、成功 smoke、可行的定向失败注入及完整门禁，给出带证据的 `PASS` 或 `FAIL`。
6. 若为 `FAIL`，列出具体文件/控制流、风险和最小修正条件；由实施终端修改，再由同一 reviewer 重跑受影响检查并更新最终结论。

## 7. CI 与完成定义

本轮只有满足以下全部条件才可结束：

- 实现者验证和同一独立 reviewer 最终 `PASS`，审核报告包含成功路径、失败路径、性能阶段和清理证据。
- 普通提交推送后，等待该准确 commit 对应的 GitHub Actions；`openapi`、`frontend`、`backend`、`e2e`、`container-smoke`、`security`、`release-candidate` 所有 required/dependent jobs 全部成功。
- `container-smoke` 日志能从阶段标记定位执行进度；若仍失败，必须依据新诊断回到实施—同一 reviewer 复审循环，不能用 `82d0f89` 的旧绿色 run 替代。
- 将最终 commit SHA、CI run ID/URL、各 job 结果及 Round 08 reviewer 结论记录到 AI change log/交接证据；证据补录也需轻量独立复核并等待其提交 CI。
- 未放宽任何业务或性能断言，仓库无 secret、临时文件和无关修改；`TODO.md` 的“发布 MVP”仍保持未勾选，tag/Release 继续等待明确人工授权。
